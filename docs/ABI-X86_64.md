# Zuki x86_64 Application Binary Interface — Implementer Specification v1.0

**Status:** Canonical  
**Authority:** Subordinate to *Zuki Abstract ABI v1.0-Canonical* and *Zuki System Architecture v1.0-Canonical*  
**Scope:** Binding for all x86_64 code under `/kernel/entry`, `/kernel/sys`, and any x86_64 userspace component that invokes syscalls or crosses the user/kernel boundary.

This document maps the abstract ABI onto x86_64 and defines:

- syscall entry and return mechanism
- concrete register assignments for logical slots
- stack and frame invariants
- clobber and preservation rules
- capability wire handling
- user/kernel memory-boundary rules

If code and this document disagree, the code is wrong.

---

## 1. Entry and return on x86_64

### 1.1 Entry instruction

- Syscalls enter the kernel via the `syscall` instruction in 64-bit mode.
- `sysenter`, `int 0x80`, and all other mechanisms are not part of the canonical v1.0 ABI.

### 1.2 Privilege transition

- `syscall` transitions from CPL3 to CPL0 using MSR-configured entry points.
- The kernel must configure `STAR`, `LSTAR`, and `SFMASK` consistently with this document.
- If `GS` base switching is used, it must be performed explicitly in the entry path. It is not implied by the abstract ABI.

### 1.3 Entry-state invariants

On x86_64 `syscall` entry:

- `rax` contains the syscall number
- `rdi`, `rsi`, `rdx`, `r10`, `r8`, and `r9` contain the logical argument slots
- `rcx` contains the user return RIP captured by hardware
- `r11` contains the masked user RFLAGS captured by hardware

The kernel must treat `rcx` and `r11` as part of the architectural syscall-entry contract.

### 1.4 Return path

- The fast return path uses `sysretq` when the return state satisfies the x86_64 architectural constraints for safe `sysretq` return.
- If those constraints are not satisfied, the kernel must use a correct alternate return path defined by the x86_64 entry implementation.

The ABI requires correct return to userspace. It does not require `sysretq` in cases where the architecture makes that unsafe or invalid.

### 1.5 Return-state sanitization

Before returning to userspace, the kernel must ensure:

- return RIP is valid for user mode
- return RFLAGS is sanitized according to architectural and security rules
- no kernel privilege state leaks into user-visible execution state

## 2. Logical slots to x86_64 registers

### 2.1 Syscall number

- Logical syscall number is passed in `rax` on entry.
- On return, `rax` carries `ret0`, the primary return value or error code.

### 2.2 Argument slots

Logical argument slots map to registers as follows:

- `arg0` → `rdi`
- `arg1` → `rsi`
- `arg2` → `rdx`
- `arg3` → `r10`
- `arg4` → `r8`
- `arg5` → `r9`

No additional syscall arguments are implicitly read from the user stack by the kernel.

### 2.3 Return slots

- `ret0` → `rax`
- `ret1` → `rdx` when and only when the syscall contract explicitly defines a secondary return value

If a syscall does not define `ret1`, `rdx` is caller-clobbered.

---

## 3. Clobber and preservation rules

### 3.1 Caller-clobbered across syscall

Userspace must treat the following registers as volatile across a syscall:

- `rax`
- `rcx`
- `rdx`
- `rsi`
- `rdi`
- `r8`
- `r9`
- `r10`
- `r11`

This includes `rcx` and `r11`, which are architecturally clobbered by `syscall`.

### 3.2 Preserved across syscall return

The kernel must preserve and restore:

- `rbx`
- `rbp`
- `r12`
- `r13`
- `r14`
- `r15`

The kernel must also preserve the user-visible architectural execution contract, including:

- user `rsp`
- required user-visible floating-point or SIMD state according to the selected x86_64 state-management mechanism

Any chosen XSAVE/FXSAVE policy must preserve the architectural contract visible to userspace. “Lazy” or optimized handling must not weaken correctness.

## 4. Stack and frame invariants

### 4.1 User stack on entry

On syscall entry:

- `rsp` is a user-mode stack pointer value
- the kernel must treat it as untrusted user state
- the kernel must not dereference user stack memory without explicit validation

The abstract ABI does not require the kernel to consume syscall arguments from the stack.

### 4.2 Alignment

User code must obey the userspace calling convention and stack-alignment rules expected by the x86_64 userspace ABI it targets.

The kernel must not assume any additional stack layout beyond what is required to preserve correct userspace execution and validated user-memory access.

### 4.3 Kernel stack

- Each thread has a kernel stack configured by the entry and scheduler code.
- The entry path must switch to the correct kernel stack before calling the common syscall handler.
- The kernel must never expose kernel stack addresses or contents to userspace.

## 5. Capability wire format on x86_64

### 5.1 CapWire layout

The canonical `CapWire` structure on x86_64 is:

```c
typedef struct CapWire {
    uint64_t object_id;
    uint64_t gen_at_mint;
    uint64_t rights;
} CapWire;
````

Requirements:

* field order is fixed
* each field is 64-bit
* layout must remain stable for the ABI version

### 5.2 Passing capabilities

When a syscall takes a capability argument, the syscall contract must define whether it is passed as:

* a user pointer to a `CapWire`, or
* an inline encoding built from syscall argument slots

If a pointer form is used, the kernel must:

* treat the pointer as a user virtual address
* validate bounds
* copy the `CapWire` into kernel memory
* reconstruct a logical capability
* validate it via `cap_use()`

Raw integers must never be interpreted as kernel object references unless the syscall contract explicitly defines an inline capability encoding consistent with the abstract ABI.

---

## 6. User-memory access rules

Any user pointer carried in syscall arguments or nested argument structures must be treated as a user virtual address only.

The kernel must:

* validate address range
* validate canonical-address form as required by x86_64
* perform explicit copy-in or copy-out
* handle faults deterministically using the syscall’s defined failure model

The kernel must not:

* dereference user pointers directly in kernel context without validation
* rely on user stack or heap layout for security decisions
* treat a faulting user address as a reason to widen authority or partially complete an operation

---

## 7. Error and return semantics on x86_64

* `rax` carries the primary return value or error code
* `rdx` carries `ret1` only when the syscall contract explicitly defines a secondary return value
* no thread-local errno or ambient error channel exists at the ABI layer

The exact success and failure encoding is defined by `SYSCALL.md` and the abstract ABI, and this x86_64 mapping must implement it exactly.

---

## 8. Blocking, abort, and restart on x86_64

The x86_64 mapping does not alter the abstract rules:

* blocking is governed by `SCHED.md`, `IPC.md`, `TIMERS.md`, and `DEVICE.md`
* abortability and restart semantics are as defined in `SYSCALL.md` and `ABI.md`

The x86_64 entry and return path must:

* not leak partial register or stack state that could be interpreted as authority
* not widen authority on restart
* not complete a syscall twice
* not allow stale return-path state to bind to a new operation

---

## 9. Forbidden patterns

On x86_64, the following are specification violations:

* using `int 0x80`, `sysenter`, or other non-canonical mechanisms as the v1.0 syscall ABI
* interpreting raw register values as kernel object references without capability reconstruction
* relying on unvalidated user register or stack contents for security decisions
* passing capabilities outside the defined `CapWire` or explicitly defined inline encoding
* using `ret1` in `rdx` when the syscall contract does not define it
* clobbering preserved registers across syscall handling
* leaking kernel stack contents, kernel register contents, or unsanitized return state back to userspace
* using `sysretq` in entry states where x86_64 architectural rules make it unsafe or invalid

Any such change is a violation, not an optimization.

## 10. Implementation checklist

Before merging any x86_64 ABI-related change:

### Entry and return

* is `syscall` used consistently as the canonical entry mechanism?
* are `STAR`, `LSTAR`, and `SFMASK` configured correctly?
* is the return path using `sysretq` only when safe, with a correct fallback otherwise?

### Registers

* do `rax`, `rdi`, `rsi`, `rdx`, `r10`, `r8`, and `r9` map correctly to logical slots?
* are `rcx` and `r11` handled according to x86_64 syscall semantics?
* are preserved registers saved and restored correctly?

### Capabilities

* is `CapWire` layout correct and stable?
* are capability arguments reconstructed and validated via `cap_use()`?

### User memory

* are user pointers validated and copied safely?
* are canonical-address checks and fault handling correct and deterministic?

### Tests

* syscall number decoding
* argument mapping correctness
* capability wire handling
* preserved-register integrity
* return-path sanitization
* error and return behavior
* interaction with `SYSCALL.md` invariants

If any answer is no, unknown, or hand-waved, the change is invalid.
